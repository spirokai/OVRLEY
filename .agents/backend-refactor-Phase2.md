# Phase 2 — Typed Errors and Typed Metrics: Detailed Implementation Plan

## Purpose

Replace `Result<T, String>` with structured `CoreError`/`CoreResult<T>` and replace stringly-typed metric keys with a `MetricKind` enum — all without changing production behavior.

---

## Prerequisites

Before starting Phase 2, Phase 1 must be complete:

- [x] All tests are in dedicated `tests/` directories (no inline `#[cfg(test)] mod tests`, no `#[path]`)
- [x] Shared test config (`tests/common/test_config.rs`) exists and is used by all tests
- [x] `cargo test` passes
- [x] No production behavior changed

If Phase 1 is incomplete, stop and complete it first. Phase 2 builds on the test safety net created in Phase 1.

### Step 2: Create `ovrley_core/src/error.rs`

**File:** `ovrley_core/src/error.rs` (new file)

```rust
use std::path::PathBuf;
use thiserror::Error;

pub type CoreResult<T> = Result<T, CoreError>;

#[derive(Error, Debug)]
pub enum CoreError {
    #[error("Invalid configuration: {0}")]
    Config(String),

    #[error("Activity parse error: {0}")]
    Activity(String),

    #[error("Render error: {0}")]
    Render(String),

    #[error("Encoding error: {0}")]
    Encode(String),

    #[error("IO error at {path}: {source}")]
    Io {
        path: PathBuf,
        #[source]
        source: std::io::Error,
    },

    #[error("FFmpeg error (exit {status}): {stderr}")]
    Ffmpeg {
        status: std::process::ExitStatus,
        stderr: String,
    },

    #[error("FFmpeg not found: {0}")]
    FfmpegNotFound(String),

    #[error("Render cancelled")]
    Cancelled,

    #[error(transparent)]
    Serialization(#[from] serde_json::Error),
}
```

**Keep in mind:**
- This is the **only** error enum for the entire crate during Phase 2. Do not create `ConfigError`, `RenderError`, etc. — a sub-error hierarchy is Phase 3+ material and only if justified (master plan §7.2).
- The `Display` impl (generated by `#[error(...)]`) produces user-readable strings. These will be shown to the frontend via `.to_string()` at the Tauri boundary. Ensure each message reads like a human-readable sentence.
- `Io` includes the file path for diagnostics — this is required by master plan §7.3.
- `Ffmpeg` includes exit status and stderr for diagnostics.
- `Cancelled` is a deliberate signal, not a real error — but it is treated as a `Result` error for control flow. Its display message must match the existing `"Render cancelled"` string that cancellation detection strings currently check for (master plan §3.2 references `error.to_lowercase().contains("cancelled")` in `commands/mod.rs:209`).
- `Serialization` uses `#[from]` so `?` operator works directly on `serde_json::Error`.
- No `Other(String)` catch-all — if a variant is temporarily needed during migration, mark it with `// TODO: migrate to typed variant` (master plan §7.4).

```rust
    Temperature,
}
```

**Verification:**
```bash

```

**Keep in mind:**
- **Verify frontend JSON compatibility before finalizing this list.** Run a full-text search for metric string literals across the entire codebase to confirm this is the complete set. Missing a metric breaks deserialization. Extra metrics are harmless but add dead code.
- The serde `rename` values must be **exact** matches for the strings currently used in:
  - `config/mod.rs` match arms in `render_data_requirements` (currently `"speed"`, `"heartrate"`, `"elevation"`, `"time"`, `"gradient"`, `"cadence"`, `"power"`, `"temperature"`)

  - `render/widgets/value.rs` match arms

- `PartialEq, Eq, Hash` are needed for `HashMap<MetricKind, ...>` usage.
- `Copy` is intentional — `MetricKind` is a small enum and will be passed by value in render hot paths (avoids unnecessary heap allocation, master plan §2.7).

### Step 5: Add Serde Round-Trip Tests for `MetricKind`

**File:** `ovrley_core/tests/metric_kind_serde_tests.rs` (new file)

```rust
use ovrley_core::MetricKind;
use serde_json;

#[test]
fn metric_kind_round_trip_all_variants() {
    let cases = vec![
        (MetricKind::Speed, r#""speed""#),
        (MetricKind::Heartrate, r#""heartrate""#),
        (MetricKind::Elevation, r#""elevation""#),
        (MetricKind::Time, r#""time""#),
        (MetricKind::Gradient, r#""gradient""#),
        (MetricKind::Cadence, r#""cadence""#),
        (MetricKind::Power, r#""power""#),
        (MetricKind::Temperature, r#""temperature""#),
    ];

    for (kind, expected_json) in cases {
        let serialized = serde_json::to_string(&kind).unwrap();
        assert_eq!(serialized, expected_json, "serialization mismatch for {:?}", kind);

        let deserialized: MetricKind = serde_json::from_str(expected_json).unwrap();
        assert_eq!(deserialized, kind, "deserialization mismatch for {:?}", kind);
    }
}

#[test]
fn unknown_metric_deserialization_fails() {
    let result: Result<MetricKind, _> = serde_json::from_str(r#""unknown_metric""#);
    assert!(result.is_err(), "Unknown metric should fail deserialization");
}
```

### Step 8: Migrate Config Module Errors — Execution

**Target files:** `ovrley_core/src/config/mod.rs`

**Process for each function:**

1. Change return type from `Result<T, String>` to `CoreResult<T>`
2. Add `use crate::error::{CoreError, CoreResult};` at top of file
3. Replace `Err("message".to_string())` with `Err(CoreError::Config("message".into()))`
4. Replace `Err(format!("message: {detail}"))` with `Err(CoreError::Config(format!("message: {detail}")))`
5. Do NOT change the error message text — only wrap it

**Example transformation:**

```rust
// Before:
pub fn parse_config_json(input: &str) -> Result<RenderConfig, String> {
    let raw: RawRenderConfig = serde_json::from_str(input)
        .map_err(|e| format!("Failed to parse config JSON: {e}"))?;
    // ...
}

// After:
pub fn parse_config_json(input: &str) -> CoreResult<RenderConfig> {
    let raw: RawRenderConfig = serde_json::from_str(input)
        .map_err(|e| CoreError::Config(format!("Failed to parse config JSON: {e}")))?;
    // ...
}
```

For `serde_json::Error` (which has a `#[from]`), you can also use:

```rust
// Alternative using the From impl:
pub fn parse_config_json(input: &str) -> CoreResult<RenderConfig> {
    let raw: RawRenderConfig = serde_json::from_str(input)?;
    // This auto-converts serde_json::Error -> CoreError::Serialization
    // BUT: loses the "Failed to parse config JSON" prefix.
    // If that prefix was visible to the frontend, use map_err instead.
}
```

**Keep in mind:**
- When `?` operator is used on `serde_json::Error`, it converts to `CoreError::Serialization` via the `#[from]` impl. If the original error had a custom prefix like `"Failed to parse config JSON: ..."`, that prefix is lost. **Preserve the prefix if it was user-visible.** The `#[from]` shortcut is only safe when the raw serde error message is sufficient.
- After migration, verify the error display strings match the old ones:
  ```rust
  // Old: Err("Invalid configuration: missing fps".to_string())
  // New: Err(CoreError::Config("Invalid configuration: missing fps".into()))
  // Display: "Invalid configuration: missing fps"  (note: "Invalid configuration: " is prepended by CoreError::Config)
  ```
  Wait — this is a problem! The old error manually said `"Invalid configuration: ..."` and the new `CoreError::Config` ALSO prepends `"Invalid configuration: "`. The result would be:
  ```
  "Invalid configuration: Invalid configuration: missing fps"
  ```
  **This is a silent behavior change.** Every error message that already contains the prefix `"Invalid configuration: "` in the inner string must be adjusted:
  ```rust
  // Old inner string: "Invalid configuration: missing fps"
  // New: CoreError::Config("missing fps".into())
  // Display: "Invalid configuration: missing fps" ✓
  ```

  **Do a complete audit of every error message string in config before writing the migration.** Check whether the old message already contains the prefix that `CoreError` variant will add.

**Verification:**
```bash
cargo check -p ovrley_core
cargo test -p ovrley_core
```

Manual verification: check that existing config fixtures still parse without error string changes.

- **`ffmpeg_composite_profiles.rs`** currently returns `Option<CompositeProfile>` (master plan §17.15) — this is not a `Result` and should be migrated to `CoreResult` during this step. Use `CoreError::Encode("Unknown composite profile: {name}".into())` or `CoreError::Config(...)` depending on whether profile resolution is considered an encode or config concern. Read the file to decide.
- **`video_composite_debug.rs`** uses `Result<T, String>` — migrate to `CoreResult` using `CoreError::Encode(...)` or `CoreError::Io { ... }` as appropriate (master plan §17.16).
- **`progress.rs`** has inline `#[cfg(test)] mod tests` at line 106 (master plan §17.14) — these should have been moved in Phase 1. If they are still there, note it as a Phase 1 deficiency but do NOT fix it in Phase 2. Focus only on error type migration.

### Step 13: Update Tauri Boundary (`src-tauri/src/lib.rs`)

**Target file:** `src-tauri/src/lib.rs`

**Process:** Tauri commands must return `Result<String, String>` (the Tauri IPC contract). Convert `CoreResult` to `Result<String, String>` at the boundary:

```rust
// Pattern for each Tauri command:
#[tauri::command]
async fn backend_render(
    state: tauri::State<'_, AppState>,
    // ... params
) -> Result<String, String> {
    commands::backend_render(/* ... */)
        .map(|value| serde_json::to_string(&value).map_err(|e| e.to_string()))
        .map_err(|e| e.to_string())
        .and_then(|inner| inner)
        // Or more clearly:
        // .map_err(|e| e.to_string())
        // .and_then(|json| json.map_err(|e| e.to_string()))
}
```

## Blockers and Known Risks

### Risk 1: Error Message Double-Wrapping
The `CoreError` variants automatically prepend a prefix (e.g., `"Invalid configuration: "`). If the original error string already contains this prefix, the frontend will see a doubled message. **Mitigation:** audit every error string during migration and strip redundant prefixes.

### Risk 2: Cancellation Detection Breakage
The string `"cancelled"` in error messages is used for control flow. If the migration inadvertently changes this string, cancellation silently breaks. **Mitigation:** the error display test in Step 6 explicitly validates this. Also replace string matching with pattern matching on `CoreError::Cancelled` in Step 12.

### Risk 3: Unknown Metric Deserialization Now Fails
Currently, unknown metric keys are silently ignored (match arm `_ => {}` pattern in multiple files). After migration, an unknown metric will fail at the serde deserialization boundary. This is the correct behavior (silent ignoring is a bug) but may surface previously hidden bad data.

**Intentional behavior change (per master plan §2.1):**
```
Intentional behavior change:
- Previous behavior: Unknown metric keys in config JSON silently matched the
  wildcard arm (_ => {}) and were ignored with no error.
- New behavior: Unknown metric keys cause a serde deserialization error at
  the MetricKind boundary, returning a clear error to the frontend.
- Reason: Silent ignoring of unknown metrics is a defect — it hides typos
  and misconfigured templates from the user. Failing loudly is safer.
- Tests added: metric_kind_serde_tests::unknown_metric_deserialization_fails
- Manual validation: Submit a template with "metric": "unknown_field" and
  confirm the frontend receives a readable error instead of silent success.
```

### Risk 4: Hot Path Regression in Render/Encode
Adding `CoreError::Render(...)` wrapping inside frame loops could add allocation overhead. **Mitigation:** only wrap errors at function boundaries, not per-frame. Profile before/after if uncertain.

### Risk 5: Phase 1 Incompleteness
If Phase 1 did not fully migrate all inline tests, the `#[cfg(test)]` modules in `render/format.rs`, `encode/progress.rs`, and others will still exist. These tests may reference old error types and fail. **Mitigation:** if Phase 1 is incomplete, note it but do the minimum necessary to keep tests passing. Do not expand Phase 2 scope to finish Phase 1 work.
